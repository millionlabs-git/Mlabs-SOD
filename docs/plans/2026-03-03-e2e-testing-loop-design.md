# E2E Testing Loop Design

Post-deploy interactive testing with Visual Playwright, test data seeding, transactional email verification via Resend, and an automated fix-retest loop.

## Problem

The pipeline builds and deploys apps but key features are broken on the live site. The current reviewer agent only does code-level review and static VP screenshots — no interactive testing of actual user flows. There is no mechanism to verify that forms submit, APIs respond, emails send, or navigation works.

## Decision Record

- **Test timing:** After deploy only (not per-task or pre-deploy)
- **Seed timing:** During deploy, before E2E tests
- **Seed accounts:** Verified account per user type
- **Email provider:** Resend (API key on worker, shared across apps)
- **Approach:** Dedicated E2E Tester Agent with fix-retest loop

## Pipeline Change

```
Before: Clone → Plan → Scaffold → Build → Deploy → Review → PR
After:  Clone → Plan → Scaffold → Build → Deploy → Seed → E2E Test → Fix Loop → PR
```

Review phase remains but focuses on code quality only. Functional correctness is handled by the E2E tester.

## 1. USER_FLOWS.md Manifest

Generated by the architect agent during planning. Covers every user-facing feature for every user type. Each flow is a complete executable test script with exact VP commands.

### Format

```markdown
# User Flows

## Meta
app_type: <type>
user_types:
  - role: <role>
    seed_account: <email> / <password>
    description: <what this user does>

## Flow: <flow-id>
priority: critical | high | medium
user_type: <role>
depends_on: [<flow-ids>]
preconditions:
  - <required state>
steps:
  - action: goto | fill | click | wait | screenshot | select | check_email | assert
    <action-specific params>
postconditions:
  - <expected state after flow>
```

### Action Types

**goto** — Navigate to a URL.
```yaml
- action: goto
  url: /login
  expect_visible: "Sign in"  # quick sanity check page loaded
```

**fill** — Type into an input. Multiple comma-separated selectors tried left-to-right.
```yaml
- action: fill
  selector: input[name="email"], input[type="email"], input[placeholder*="email" i]
  value: tenant@test.mlabs.app
```

**click** — Click an element.
```yaml
- action: click
  selector: button[type="submit"], button:has-text("Save"), button:has-text("Create")
```

**wait** — Wait for a condition. Supports `visible`, `navigation`, and `any` (multiple conditions).
```yaml
- action: wait
  for: any
  conditions:
    - { type: visible, text_contains: "created" }
    - { type: navigation, url_contains: "/buildings" }
    - { type: visible, selector: "text=Test Building Alpha" }
  timeout: 5000
```

**select** — Select from a dropdown.
```yaml
- action: select
  selector: select[name="priority"]
  value: medium
  fallback: skip  # optional field, skip if not present
```

**screenshot** — Capture current state.
```yaml
- action: screenshot
  name: building-form-filled
```

**check_email** — Read a sent email via Resend API. Polls until email arrives.
```yaml
- action: check_email
  to: tenant@test.mlabs.app
  subject_contains: "reset", "password"
  timeout: 15000
  extract: reset_link  # extracts first URL containing /reset or /password
```

**assert** — Verify page state.
```yaml
- action: assert
  type: visible | no_error | email_received
  selector: h1
  text_contains: dashboard
  check: console  # for no_error type
```

### Selector Strategy

Each step has comma-separated CSS selectors. The tester tries each left-to-right and uses the first match. This handles apps using `data-testid`, semantic HTML, or text-based selectors. If none match, the step fails with "element not found" and a list of what IS on the page.

### Flow Dependencies

`depends_on` defines ordering. If `tenant-login` fails, all flows with `depends_on: [tenant-login]` are marked BLOCKED rather than attempted.

### Expected Coverage (example: property management app)

**Tenant (10+ flows):** login, logout, register, password reset, set display name, view dashboard, view lease, submit maintenance request, view request status, edit/cancel request, update profile, notification preferences.

**Landlord (12+ flows):** login, create/edit/delete building, add/edit/delete unit, assign tenant, view/respond to maintenance requests, mark resolved, view tenant list, invite tenant, financial summary.

**Admin (6+ flows):** login, view all users, manage accounts, view all buildings, system settings, audit log.

**Cross-cutting (4+ flows):** unauthenticated redirect, invalid credentials error, session expiry, mobile viewport check.

The architect generates ALL flows — 30-40+ for a typical app.

  address: 123 Test Street, San Francisco, CA 94102

  units:

### Seeder Implementation

A standalone Claude agent call after deploy. NOT a fixed SQL script — adapts to whatever schema/ORM the app uses.

**Seeder prompt:**
```
You have a deployed app at {url} with database at {database_url}.
Seed the following test data by writing and executing a seed script.

Read the app's schema/models to understand the exact table structure.
Hash passwords using the app's own auth library (bcrypt, argon2, etc.).
Set email_verified=true (or equivalent) on all accounts.
Use the app's actual ORM/query patterns.
Run the script against the production DATABASE_URL.
Verify each account exists by querying the users table.
```

The seeder reads the codebase, writes a one-off seed script, executes it against the live DB, and verifies.

## 3. Tester Agent

A new 6th subagent in the pipeline, same pattern as architect/scaffolder/builder/fixer/reviewer.

### Inputs

- USER_FLOWS.md (full manifest)
- SEED_DATA.md (credentials and test data)
- Deployed app URL
- Visual Playwright MCP server
- Resend API key (for email checking)

### Behavior

1. Runs every flow in dependency order
2. For each flow: execute steps sequentially using VP
3. On step failure: screenshot, record error, skip remaining steps in flow (mark BLOCKED)
4. **Never stops on failure** — runs ALL flows
5. After each action, checks browser console for errors
6. Produces TEST_REPORT.md

### Selector Resolution

For each step's comma-separated selectors:
1. Try each selector left-to-right
2. Use the first one that exists on the page
3. If none match: FAIL with "element not found" + list of elements actually present

### Email Checking

For `check_email` steps:
1. Poll Resend API (`GET /emails`) filtered by recipient
2. Wait up to `timeout` ms for matching email
3. Extract URLs from email HTML body matching the `extract` pattern
4. Store extracted URL as a variable for subsequent `goto` steps

### TEST_REPORT.md Output

```markdown
# E2E Test Report

## Summary
total_flows: 34
passed: 28
failed: 4
blocked: 2
run_time: 12m 34s
app_url: https://app.fly.dev

## Results

### PASS: tenant-login (8.2s)
All 7 steps passed.

### FAIL: landlord-create-building (12.1s)
Failed at step 4: fill input[name="address"]
  error: Element not found
  selectors_tried: input[name="address"], input[placeholder*="address" i]
  page_state: Form visible but only has fields: name, city, state, zip
  screenshot: screenshots/landlord-create-building-step4-fail.png
  diagnosis: Building form missing address field in BuildingForm component.

### BLOCKED: landlord-view-maintenance-requests
  reason: Depends on tenant-submit-maintenance-request which failed
  recommendation: Could use seeded maintenance request data instead

## Failed Flow Details
- landlord-create-building: Missing address field in form component
- tenant-submit-maintenance-request: POST /api/maintenance returns 500
- password-reset-full: Reset email sent but link returns 404 (/reset route not implemented)
- tenant-edit-profile: Display name saves but doesn't persist (not written to DB)
```

The tester diagnoses failures — identifies likely cause by examining page state, console errors, and network responses. This gives the fixer agent actionable information.

## 4. Fix-Retest Loop

```
              ┌──────────────────────────────┐
              │                              │
              ▼                              │
        ┌──────────┐                         │
        │  Tester   │                        │
        │  Agent    │                        │
        └────┬─────┘                         │
             │                               │
             ▼                               │
      TEST_REPORT.md                         │
             │                               │
             ▼                               │
    ┌────────────────┐     All pass?         │
    │  Check results ├──────────────► Done   │
    └────────┬───────┘                       │
             │ Has failures                  │
             ▼                               │
        ┌──────────┐                         │
        │  Fixer    │                        │
        │  Agent    │                        │
        └────┬─────┘                         │
             │                               │
             ▼                               │
        ┌──────────┐                         │
        │ Redeploy  │                        │
        └────┬─────┘                         │
             │                               │
             └───────────────────────────────┘
                  (max 5 iterations)
```

### Loop Mechanics

1. **Tester runs** → produces TEST_REPORT.md
2. **All pass** → move to PR creation
3. **Failures exist** → fixer agent gets TEST_REPORT.md + codebase
4. **Fixer fixes ALL issues** in a single pass (not one at a time)
5. **Commit + redeploy** to Fly.io
6. **Tester re-runs** — only previously failed/blocked flows + 3 critical smoke flows
7. **Repeat** up to 5 iterations
8. **Still failing after 5** → create PR anyway with test report attached, noting remaining failures

### Retest Optimization

On iteration 2+, the tester runs:
- All previously failed flows
- All previously blocked flows (dependency might now pass)
- 3 critical smoke flows (login, one create, one view) to catch regressions

This cuts retest from ~12 minutes to ~4 minutes.

### Fixer Prompt

```
Fix ALL of the following issues. Do not fix just one and stop.

{TEST_REPORT.md failed flow details}

Fix the root cause, not the symptom.
If a missing form field is reported, check schema vs component and add it.
If an API returns 500, read the route handler and fix the error.
Run the app's own tests after fixing to check for regressions.
Commit: "fix: resolve E2E test failures (iteration N)"
```

## 5. Transactional Email Testing via Resend

### Setup

- `RESEND_API_KEY` is set on the Cloud Run worker (done)
- Worker passes it to deployed apps as a Fly secret
- Tester agent uses the same key to read sent emails
- Architect agent is told: "Use Resend for transactional emails. `RESEND_API_KEY` is available."

### Email-Dependent Flows

**Password reset:**
1. Trigger reset from UI (fill email, click send)
2. `check_email` polls Resend API for reset email
3. Extract reset link from email HTML
4. Navigate to reset link in browser
5. Fill new password, submit
6. Verify login with new password works

**Registration + email verification:**
1. Fill registration form, submit
2. `check_email` polls for verification email
3. Extract verification link
4. Navigate to link
5. Verify account is now verified (can log in)

**Tenant invite:**
1. Landlord sends invite from UI
2. `check_email` polls for invite email
3. Extract invite link
4. Navigate to link, complete account setup
5. Verify new tenant can log in

### Test Email Addresses

All test accounts use `@test.mlabs.app` domain. Resend handles delivery. The `newuser@test.mlabs.app` address is NOT pre-seeded — used for registration flow testing.

## 6. Pipeline Integration Points

### Architect Agent (Plan Phase)

Additional instruction:
```
Generate USER_FLOWS.md covering every user-facing feature for every user type.
Generate SEED_DATA.md with test accounts (verified, one per user type) and
enough data for flows to execute (buildings, units, relationships).
Use Resend for transactional emails (RESEND_API_KEY available).
```

### Deployer (Deploy Phase)

No changes. Returns app URL after deploy.

### New: Seeder Step

After deploy, before E2E tests. Standalone agent call that reads SEED_DATA.md and the app's codebase, writes a seed script, runs it against the live DB.

### New: Tester Agent

6th subagent. Executes USER_FLOWS.md against live URL using VP. Produces TEST_REPORT.md.

### New: Fix-Retest Loop

Orchestrator logic: tester → fixer → redeploy → tester, up to 5 iterations.

### Reviewer Agent

Unchanged. Focuses on code quality, security, architecture. No longer responsible for functional testing.

### PR Creation

TEST_REPORT.md included in PR body. Shows what was tested and results.
